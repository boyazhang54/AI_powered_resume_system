## 1. 背景与问题

在招聘流程中，HR 和招聘负责人经常需要在短时间内处理大量 PDF 简历。传统人工筛选存在以下问题：

- 简历数量大，人工阅读成本高。
- 候选人表达方式不统一，同一能力可能有多种写法。
- JD 与简历之间的匹配判断依赖个人经验，结果不够稳定。
- 缺少结构化分析结果，难以快速比较候选人。
- 对技术岗位而言，技能关键词、项目经验、工作年限等信息分散在简历文本中，人工提取效率低。

本产品希望通过 PDF 解析、AI 信息抽取、语义匹配评分和可视化展示，帮助招聘方快速完成初筛，提高简历筛选效率和一致性。

## 2. 产品目标

### 2.1 业务目标

- 将单份简历初筛时间从人工 5-10 分钟降低到 1 分钟内。
- 帮助招聘者快速识别候选人与岗位 JD 的匹配程度。
- 输出结构化、可解释的分析结果，而不是仅给出简单分数。
- 为后续批量简历筛选、候选人排序、招聘数据沉淀打基础。

### 2.2 用户目标

- HR 可以快速上传简历并获得候选人基本信息。
- 招聘负责人可以输入岗位 JD，查看匹配分数和缺失技能。
- 技术面试官可以查看 AI 给出的语义匹配证据，辅助判断候选人是否值得进入下一轮。
- 用户可以查看历史匹配记录，避免重复分析。

### 2.3 MVP 成功标准

- 支持用户登录和注册。
- 支持上传单个 PDF 简历。
- 能提取简历文本并结构化展示关键信息。
- 支持输入岗位 JD 并生成匹配评分。
- 能展示多维度雷达图、缺失技能和语义匹配证据。
- 能保存并查看当前用户的历史匹配记录。
- 前端和后端均可公网访问并完成完整流程。

## 5. 核心用户流程

### 5.1 首次使用流程

```txt
打开系统
  -> 注册账号
  -> 登录成功
  -> 进入简历分析页面
```

### 5.2 简历分析流程

```txt
上传 PDF 简历
  -> 后端解析 PDF 文本
  -> AI/规则提取简历信息
  -> 前端展示结构化信息
  -> 用户粘贴岗位 JD
  -> 点击开始匹配
  -> 后端进行 AI 语义匹配和规则评分
  -> 前端展示评分、雷达图、缺失技能、证据
  -> 系统保存历史记录
```

### 5.3 历史记录流程

```txt
用户点击历史记录
  -> 系统查询当前用户历史
  -> 展示候选人姓名、联系方式、岗位、匹配分数、时间
```

## 6. 功能需求

### 6.1 用户登录与注册

#### 需求描述

用户需要登录后才能使用简历解析、岗位匹配和历史记录功能。

#### 功能规则

- 用户名长度不少于 3 位。
- 密码长度不少于 6 位。
- 用户名不可重复。
- 登录成功后返回访问 Token。
- 前端将 Token 保存到 `localStorage`，后续请求自动携带。
- 密码使用哈希存储，不保存明文。

#### 接口

```http
POST /api/auth/register
POST /api/auth/login
GET  /api/auth/me
```

### 6.2 简历上传与解析

#### 需求描述

用户可以上传单个 PDF 简历，系统提取 PDF 文本并清洗。

#### 功能规则

- 仅支持 PDF 文件。
- 支持多页 PDF。
- 空文件或无法提取文本时返回错误提示。
- 同一 PDF 重复上传时可命中缓存。

#### 输出内容

- `resume_id`
- 原始文本
- 清洗后的文本
- 结构化简历信息

#### 接口

```http
POST /api/resumes/parse
```

### 6.3 简历关键信息提取

#### 需求描述

系统从简历文本中提取候选人关键信息。

#### 字段范围

| 类别 | 字段 | 优先级 |
|---|---|---|
| 基本信息 | 姓名、电话、邮箱、地址 | P0 |
| 求职信息 | 求职意向、期望薪资 | P1 |
| 背景信息 | 工作年限、学历背景、项目经历 | P1 |
| 技能信息 | 技能关键词 | P0 |

#### 提取策略

- 优先使用 AI 模型结构化抽取。
- AI 调用失败时使用规则兜底。
- 电话、邮箱等强格式字段使用正则辅助提取。

### 6.4 岗位 JD 输入与解析

#### 需求描述

用户可粘贴岗位 JD 文本，系统用于后续匹配评分。

#### 当前支持

- JD 文本输入。
- 示例 JD 一键填入。

#### 预留能力

- JD URL 输入。
- 网页正文抓取。
- JD 结构化解析。

### 6.5 简历匹配评分

#### 需求描述

系统根据简历内容和岗位 JD 输出综合匹配分数。

#### 评分维度

| 维度 | 说明 | 权重 |
|---|---|---|
| 技能匹配 | 候选人技能与岗位要求的匹配程度 | 35% |
| 经验相关 | 工作年限、岗位经历、职责深度是否匹配 | 25% |
| 项目相关 | 项目经历是否能证明岗位所需能力 | 20% |
| 学历匹配 | 学历或专业是否满足要求 | 10% |
| 关键词覆盖 | JD 关键能力项覆盖程度 | 10% |

#### AI 语义能力

AI 不仅做关键词匹配，还需要识别语义相近表达：

```txt
AI API
大模型接口
LLM API
模型服务调用
```

这些表达可归一为：

```txt
AI API integration
```

#### 输出内容

- 综合分数
- 五维雷达图数据
- 已匹配关键词
- 缺失技能
- 缺失技能重要性
- 逐条 JD 要求匹配证据
- AI 总结

### 6.6 历史记录

#### 需求描述

用户完成一次匹配后，系统保存历史记录，方便后续查看。

#### 保存字段

| 字段 | 说明 |
|---|---|
| 姓名 | 从简历中提取 |
| 联系方式 | 电话 / 邮箱 |
| 岗位 | 从 JD 中由 AI 或规则提取 |
| 匹配分数 | 本次分析综合分 |
| 分析模式 | AI 语义增强 / 规则兜底 |
| 时间 | 创建时间 |

#### 接口

```http
GET /api/history
```

## 7. 非功能需求

### 7.1 性能

- 单份 PDF 解析应在数秒内完成。
- AI 匹配耗时取决于模型响应，MVP 可接受 5-30 秒。
- 相同简历和 JD 重复分析时优先使用缓存。

### 7.2 可用性

- AI 调用失败时不能阻断主流程，应自动回退到规则评分。
- 上传失败、认证失败、解析失败时前端需要展示明确错误提示。

### 7.3 安全

- API Key 不提交到 GitHub。
- `.env`、数据库文件、缓存文件不进入仓库。
- 密码哈希存储。
- 前端不保存云厂商 AccessKey。

### 7.4 可扩展性

- AI 客户端采用 OpenAI-compatible 抽象，便于切换 DeepSeek、Qwen、OpenAI 等模型。
- 缓存层预留 Redis 接口。
- 数据库当前为 SQLite，后续可迁移到 RDS。

## 10. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| AI API 调用失败 | 无法生成语义评分 | 自动回退规则评分 |
| PDF 为扫描件 | 无法提取文本 | 后续接入 OCR |
| FC 临时存储丢失 | SQLite 历史记录可能丢失 | 生产环境迁移 RDS |
| JD 表达不标准 | 匹配结果不稳定 | 使用 AI 语义归一化和证据输出 |
| GitHub Pages 跨域 | 前端无法请求后端 | 配置 FC `ALLOWED_ORIGINS` |
